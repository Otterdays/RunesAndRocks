# Improvement Notes — 25 Feb 2026

Observations and ideas from a pass over the repo. Only things that are real and verifiable in the codebase or docs.

## Security & Robustness (from audit docs)

- **Admin dashboard has no auth** (06-security.md, 09-technical-debt TD-013). Admin is bound to 127.0.0.1, so risk is low unless you later expose it. If you ever bind admin to 0.0.0.0 or put it behind a reverse proxy, add at least basic auth or a shared secret.
- **PacketCodec has no max payload length** (09-technical-debt TD-017). `read()` does `ByteArray(length)` with client-supplied `length`. A malicious client can send a huge length and cause OOM. Cap payload size (e.g. 64 KB) and reject larger.
- **MoveRequest velocity is unbounded** (06-security, 09-technical-debt TD-003). Client can send very large dx/dy. MovementSystem applies it directly. Clamp in GameServer when handling the packet or in MovementSystem.
- **Username is not trimmed or length-checked** (TD-020). Empty or whitespace can create a player. Trim and reject empty before calling `loginPlayer`.
- **Duplicate LoginRequest** (TD-019). Sending login twice before spawn can orphan the first entity. Reject login if `conn.entityId != null` (already logged in).

## API & Admin Behavior

- **`/api/health`** calls `RedisFactory.getClient()` inside a try/catch. If DB failed first in ServerLauncher, Redis never inits, so `getClient()` throws. The catch returns `redis: false`. So health is safe when Redis (or DB) never initialized.
- **World reload** (reload-world action) reads from classpath resource `/world.json` via `getResourceAsStream("/world.json")`. So it reloads the bundled resource, not an arbitrary file on disk. If the intent is "edit world.json on disk and hot-reload", you'd need a configurable path or a path under a known dir and read with `File(path).readText()` (and only if you're okay with that in production).
